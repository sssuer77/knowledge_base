## 摘要

随着产品技术文档数量不断增长，传统关键词检索难以满足自然语言问答的需求。本课程项目设计并实现了一套**多路重排智能智库系统**，面向商品类说明书、规格书等文档，提供「上传即入库、提问即回答」的智能知识服务。

系统采用 **LangGraph** 编排两条核心工作流：文档导入流水线（PDF/Markdown → 解析 → 切分 → BGE-M3 向量化 → Milvus 入库）与智能查询流水线（商品确认 → 向量/HyDE/联网多路检索 → RRF 融合 → BGE 重排 → 流式生成）。后端基于 **FastAPI** 构建，前端采用 **Vue 3**，向量数据存储于 **Milvus**，会话历史存储于 **MongoDB**，文件持久化于 **MinIO**。部署采用 **Windows 宿主机 + VMware 虚拟机**混合架构，在控制成本的同时保证 GPU 本地推理性能。

项目历时 14 周，5 人团队协作完成。测试阶段共执行 42 条用例，P0 通过率 100%，验收标准 8/8 全部通过。结果表明，多路检索与重排策略能有效提升产品文档问答的准确性与覆盖面，系统稳定可用于演示与实际内网场景。

**关键词**：知识库；RAG；LangGraph；向量检索；多路重排；BGE-M3

## Abstract

This course project presents a **Knowledge Base system with multi-route retrieval and reranking** for product technical documentation. Built on LangGraph workflows, FastAPI, Milvus, and BGE-M3 hybrid embeddings, the system supports document ingestion and intelligent Q&A with streaming responses. A hybrid deployment (Windows host + VMware VM) balances cost and GPU performance. Testing achieved 100% pass rate on critical cases, validating the feasibility of the proposed architecture.

**Keywords**: Knowledge Base; RAG; LangGraph; Vector Search; Reranking; BGE-M3

## 1 绪论

### 1.1 项目背景

企业与电商平台积累了大量产品说明书、规格书、操作手册等非结构化文档。业务人员与客户常需快速查询「某型号产品如何操作」「某功能参数是多少」等问题。传统文件夹管理与关键词搜索存在以下局限：

- 无法理解自然语言问法与同义表述；
- 难以定位分散在长篇 PDF 中的具体段落；
- 通用大模型直接回答易产生**幻觉**，缺乏文档依据。

检索增强生成（RAG）通过「先检索、后生成」缓解上述问题，但单一向量检索在面对表述差异、跨段落推理时召回仍不足。本项目在 RAG 基础上引入**多路检索、融合排序与 Cross-Encoder 重排**，并针对商品类文档增加**型号识别与确认**环节，构建面向实际场景的 intelligent 知识库系统。

### 1.2 项目目标

1. 实现 PDF/Markdown 文档自动导入、切分、向量化与 Milvus 入库；
2. 实现基于商品型号确认的多路检索智能问答；
3. 提供 Web 界面上传与对话交互，支持 SSE 流式进度与回答；
4. 完成 VMware + Docker 中间件部署与 Windows 宿主机混合架构；
5. 输出完整工程文档，满足课程验收与答辩要求。

### 1.3 项目范围

**包含**：双后端服务、双前端界面、Milvus/MinIO/MongoDB 集成、会话历史、混合部署。

**不包含**：用户认证与多租户、移动端 App、自研 OCR、公网 HTTPS 部署。

### 1.4 项目组织与进度

项目周期 **2026-03-02 ～ 2026-06-08**，共 14 周，5 人团队。按「立项 → 导入子系统 → 查询子系统 → 前端联调 → 部署测试 → 文档答辩」推进。详细甘特图与任务分工见 [项目开发计划](./项目开发计划.md)。

## 3 需求分析

### 3.1 用户与场景

| 用户 | 典型场景 |
|------|----------|
| 知识库管理员 | 上传产品 PDF/MD，查看导入进度 |
| 业务咨询人员 | 自然语言提问，获取带来源的流式回答 |

### 3.2 功能需求摘要

系统划分为**文档导入**与**智能查询**两大子系统，共 22 项功能需求（详见 [需求分析规格说明书](./需求分析规格说明书.md)）。

**导入子系统**：
- 多文件上传（PDF/MD）
- PDF 经 MinerU 解析为 Markdown
- 文档切分、商品型号识别、BGE-M3 向量化
- Milvus 入库、MinIO 备份、SSE 进度推送

**查询子系统**：
- 自然语言提问、商品型号确认与查询改写
- 向量检索、HyDE 增强检索、MCP 联网搜索（三路并发）
- RRF 融合、BGE 重排、检索增强流式回答
- MongoDB 会话历史管理

### 3.3 非功能需求摘要

| 类别 | 关键指标 |
|------|----------|
| 性能 | 上传响应 ≤3s；同步问答 ≤30s；流式首 token ≤5s |
| 可靠性 | 任务失败可感知；MinIO 不可用可降级 |
| 安全 | API Key 外置 `.env`；中间件内网部署 |
| 可维护性 | 节点解耦、Prompt 外置、统一日志 |

### 3.4 用例概述

核心用例：**上传并导入文档**、**智能问答**。当商品型号模糊时系统反问用户；无匹配商品时拒绝回答并提示——避免错误检索污染答案。用例图见需求规格说明书第 4 章。

## 4 系统设计

### 4.1 总体架构

系统采用**四层逻辑架构**：表现层（Vue 前端）→ 应用层（FastAPI + LangGraph）→ 领域层（节点、模型、Prompt）→ 基础设施层（Milvus/MinIO/MongoDB/云端 API）。详细设计见 [系统设计文档](./系统设计文档.md)。

### 4.2 导入工作流设计

```
node_entry → [PDF] node_pdf_to_md → node_md_img
          → [MD]  node_md_img
→ node_document_split → node_item_name_recognition
→ node_bge_embedding → node_import_milvus
```

- 入口节点按扩展名路由 PDF/MD 路径；
- 导入节点继承 `NodeBase`，统一日志与 SSE 进度追踪；
- 入库前按 `item_name` 幂等删除旧数据。

### 4.3 查询工作流设计

```
node_item_name_confirm → [有 answer] → node_answer_output
                       → [无 answer] → 三路并发检索
                         → node_rrf → node_rerank → node_answer_output
```

- 商品确认节点：LLM 提取型号 + Milvus 对齐；
- 虚拟节点 `node_multi_search` / `node_join` 组织并发与合并；
- 答案节点流式推送 `delta` 事件至前端。

### 4.4 数据设计摘要

| 存储 | 实体 | 说明 |
|------|------|------|
| Milvus `kb_chunks` | 文档切片 + 双向量 | 混合检索主库 |
| Milvus `kb_item_names` | 商品型号 + 向量 | 查询对齐 |
| MongoDB `chat_message` | 会话消息 | 多轮上下文 |
| MinIO | PDF、图片 | 对象持久化 |
| 本地 `output/` | 任务工作目录 | 中间产物 |

完整 Schema 见 [数据设计文档](./数据设计文档.md)。

### 4.5 接口设计摘要

| 服务 | 核心接口 |
|------|----------|
| 导入 :8000 | `POST /upload`、`GET /status/{id}`、`GET /stream/{id}` |
| 查询 :8001 | `POST /query`、`GET /stream/{id}`、`GET/DELETE /history/{id}`、`GET /health` |

## 5 系统实现

### 5.1 项目结构

```
knowledge_base/
├── app/import_process/     # 导入 API + LangGraph + 前端
├── app/query_process/      # 查询 API + LangGraph + 前端
├── app/clients/            # Milvus / MinIO / MongoDB 客户端
├── app/lm/                 # BGE 嵌入、重排、LLM 封装
├── app/conf/               # 环境配置
├── prompts/                # Prompt 模板
└── docs/                   # 项目文档
```

### 5.2 关键实现说明

#### （1）LangGraph 状态驱动

导入与查询均使用 TypedDict 定义状态对象（`ImportGraphState`、`QueryGraphState`），节点间通过 state 传递数据，便于调试与扩展。

#### （2）BGE-M3 混合向量

`embedding_utils.py` 封装 BGE-M3 单例，生成 1024 维稠密向量与稀疏向量，写入 Milvus 后支持 `hybrid_search` + 加权排序。

#### （3）商品维度检索过滤

查询时在 Milvus 中使用 `item_name in [...]` 表达式过滤，缩小检索空间，显著降低跨商品干扰。

#### （4）SSE 流式通信

`sse_utils.py` 以 `session_id` / `task_id` 为键维护 Queue，FastAPI `StreamingResponse` 异步推送 progress/delta/final 事件。

#### （5）混合部署

中间件 Docker 运行于 VMware 虚拟机；Python 后端与 GPU 模型推理在 Windows 宿主机，`.env` 中通过 `<VM_IP>` 连接 Milvus/MinIO/MongoDB。

### 5.3 前端实现

- **导入页**（:8000）：文件选择与上传、`streamTaskStatus` SSE 订阅节点进度；
- **查询页**（:8001）：对话输入、`streamQuery` 流式渲染 Markdown（marked 库）。

构建产物由各自 FastAPI 服务静态托管，生产环境同源访问。

## 7 结果分析与总结

### 7.1 目标达成

项目完成了需求规格说明书中定义的全部核心功能，实现了从文档导入到智能问答的完整 RAG 闭环，达到了课程预期建设目标。

### 7.2 性能与质量

- 上传接口响应 0.8～1.6s，同步问答 12～22s，流式首 token 2.5～4.2s，均满足非功能指标；
- 15 题人工抽样评测，答案正确率 86.7%，商品识别准确率 93.3%；
- 多路检索 + 重排相比单一向量检索，在表述差异类问题上召回明显改善。

### 7.3 成本与进度

- 实际 API + 电费支出约 **¥1,340**，低于预算上限；
- 中间件里程碑延迟 2 天（端口冲突），导入里程碑延迟 3 天（MinerU 联调），均在缓冲期内完成。

### 7.4 主要问题与解决

| 问题 | 解决方案 |
|------|----------|
| Milvus 与 MinIO 端口冲突 | Milvus compose 改映射 9002/9003 |
| 商品型号无法唯一确认 | 条件路由提前返回反问/拒答 |
| VMware NAT 网络不通 | 改桥接模式 + 防火墙放行 |
| Postman 无法测 SSE | 改用 curl + Chrome DevTools |

详细分析见 [结果分析与总结](./结果分析与总结.md)。

## 8 结论与展望

### 8.1 结论

本课程项目成功设计并实现了「多路重排智能智库」系统。通过 LangGraph 工作流编排、BGE 混合向量、多路检索与 BGE 重排，在产品文档问答场景中验证了方案的可行性与实用价值。混合部署架构有效控制了基础设施成本。系统测试与验收全部通过，具备答辩演示与内网试用条件。

### 8.2 不足

- 缺少用户认证与权限管理；
- 任务状态仅存内存，服务重启后丢失；
- 未进行大规模并发压测；
- PDF 解析强依赖 MinerU 云端 API。

### 8.3 展望

1. 引入 Redis 持久化任务状态，支持断点续传；  
2. 增加 JWT 认证与多租户隔离；  
3. 使用 pytest + CI 实现自动化回归；  
4. 探索本地 MinerU 部署，降低外部依赖；  
5. 扩展知识图谱实体关系，支持结构化推理。
